# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository Nature

This repository contains the **First Principles Framework (FPF)** specification and a **Claude Code Skill** that makes FPF accessible without requiring users to know FPF terminology.

Key files:
- `FPF-Spec.md` — the monolith core specification (~59,000 lines, source of truth)
- `Readme.md` — entry point with six practical newcomer routes
- `sections/` — decomposed spec (~230 files, generated by `scripts/rebuild_all.sh`)
- `.claude/skills/fpf/SKILL.md` — Claude Code Skill entry point
- `.claude/agents/fpf-*.md` — Agent team (Classifier → Retriever → Reasoner → Reviewer)

## FPF Skill: Plain Language Contract

**FPF is invisible infrastructure.** The skill applies FPF patterns internally but NEVER exposes FPF terminology to users. Users speak their own language, answers are in their language. No "holon", "bounded context", "episteme" in output.

## What FPF Is

FPF is an "operating system for thought" — a structured framework for coordinating work across specialists, teams, and AI agents. It provides named patterns, definitions, and review rules for modeling complex work, making reasoning inspectable, and keeping decisions stable. It is **not** a methodology (like Agile), a quick-answer cheat sheet, or a linear textbook.

## Specification Structure (FPF-Spec.md)

The spec is divided into parts and clusters. Each pattern follows a numbered template: Problem Frame, Problem, Forces, Solution, Archetypal Grounding, Bias-Annotation, Conformance Checklist, Consequences, Rationale, Relations.

| Part | Name | Content |
|------|------|---------|
| **Preface** | Non-normative | Orientation essays, philosophical stance, "big storylines" |
| **A** | Kernel Architecture | Foundational ontology (holons, bounded contexts, roles), transformation engine (quartet), time/evolution, signature stack & boundary discipline (A.6.*), constitutional principles |
| **B** | Trans-disciplinary Reasoning | Aggregation algebra, emergence, trust/assurance calculus (F-G-R), canonical evolution loop, abductive reasoning |
| **C** | Kernel Extensions | Pluggable domain-specific calculi (Sys/KD/Kind/Method/LOG/CHR families), creativity/NQD, measurement, explore/exploit policy |
| **D** | Multi-scale Ethics | Ethics from agent to planetary scope, conflict topology, trust-aware mediation |
| **E** | FPF Constitution & Authoring | Vision/pillars, didactic architecture, authoring protocol, lexical law, multi-view publication (MVPK), review gates, DRR evolution governance |
| **F** | Unification Suite | Concept-sets, SenseCells, local naming, bridges, UTS (Unified Term Sheet) publication form |
| **G** | Discipline SoTA Patterns Kit | CG-Spec/CG-Frame authoring, state-of-the-art harvesting, selector/dispatcher patterns, portfolio governance |
| **H-K** | Reference | Glossary, annexes, indexes, navigation aids, lexical debt |

## Six Practical Entry Routes

When working with FPF content, the correct starting point depends on the user's **first honest burden**:

1. **Project alignment** — role/method/plan/work confusion → start with `A.1.1`, `A.15`, `B.5.1`
2. **Partly-said / language-state discovery** — cues too early to present as claims → start with `C.2.2a`, `A.16`
3. **Boundary unpacking** — mixed rules/gates/duties in boundary text → start with `A.6`, `A.6.B`, `A.6.C`
4. **Lawful comparison / selection** — honest comparison without hidden logic → start with `A.19:0`, `A.17-A.19`, `G.0`
5. **Generator / SoTA / portfolio kit** — reusable search/harvest scaffold → start with `A.0`, `G.0`, `G.1`
6. **Same-entity rewrite / explanation** — restate without changing the object of talk → start with `A.6.3.CR`, `A.6.3.RT`, `E.17.EFP`

## Key Concepts to Know

- **Holon** — entity modeled as both whole and part (System or Episteme)
- **Bounded Context** — local semantic frame with its own meanings; cross-context reuse needs explicit bridges
- **Transformer Quartet** — system bearing TransformerRole, MethodDescription, Method, Work
- **UTS** — Unified Term Sheet: structured vocabulary artifact for unstable/overloaded terms
- **DRR** — Decision Rationale Record: decision record with reasons, assumptions, open questions
- **F-G-R** — Formality, Generality (ClaimScope), Reliability triad for trust/assurance
- **MVPK** — Multi-View Publication Kit: audience-specific views of the same underlying work
- **Signature Stack** — boundary discipline that routes claims into laws, admissibility gates, deontics, and evidence
- **NQD** — Novelty-Quality-Diversity: creative generation policy
- **A.7 Strict Distinction** — Object != Description != Carrier; Role != Work

## Lexical Rules (Part K — Mandatory)

When editing normative content in the spec:
- **NEVER** use "axis" or "dimension" for measurable aspects → use **Characteristic**
- **NEVER** use "metric" as a noun in Tech register → use `U.DHCMethodRef` / `U.Measure` / Score
- **NEVER** use "applicability", "envelope", "generality", "capability envelope", "validity" as scope names → use `U.ClaimScope` (G), `U.WorkScope`, or `U.Scope`
- "point" / "data point" → **Coordinate** (on a Scale)
- "score" → **Score** (produced via a ScoringMethod)

## Working with This Repository

- The spec is too large to read in full. Use `sections/metadata.json` for instant pattern lookup.
- Pattern IDs are hierarchical: `A.6.P` is a child of `A.6`, which belongs to Part A.
- Each pattern section ends with a `### <ID>:End` marker.
- The Table of Content (lines 7-337 of FPF-Spec.md) contains a status column and dependency graph for every pattern.

## Rebuild & Sync Pipeline

Two-layer automatic update system:

**Layer 1 — GitHub Action** (`.github/workflows/rebuild-sections.yml`):
- **Hook**: push to main with FPF-Spec.md change → Python rebuild (free, no API)
- **Cron**: 1st and 15th of each month, 7:00 UTC → auto-sync fork + Python rebuild
- Pure Python: `split_spec.py` → `build_metadata.py` → `build_glossary.py` → `build_lexical.py` → `build_routes.py`

**Layer 2 — Claude Code Remote Trigger** (`trig_01P7UzjrjgsgzLpMHn84bMoo`):
- **Cron**: 1st and 15th, 9:00 Belgrade (7:00 UTC)
- Syncs upstream → Python rebuild → **AI-enhances** `_index.md` summaries and `glossary-quick.md` with plain-language descriptions
- Agent instructions: `.claude/agents/fpf-sync.md`
- Manage at: https://claude.ai/code/scheduled

**Manual rebuild** (after local changes to FPF-Spec.md):
```bash
./scripts/rebuild_all.sh
```
