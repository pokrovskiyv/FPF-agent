# Tier (ярус загрузки)

## Определение

**Tier** — стратегия загрузки секций ретривером для конкретного запроса. Три яруса: Tier 1 (по маршруту), Tier 2 (семантический откат), Tier 3 (комбинированный). Классификатор выбирает ярус по тому, насколько хорошо сообщение пользователя совпадает с известными burden; ретривер исполняет выбранную стратегию.

Ярусы нужны для баланса между качеством загрузки и вычислениями: Tier 1 самый дешёвый и точный, Tier 2 шире, но платит за инференс модели, Tier 3 использует оба.

## Как это работает в системе

| Ярус | Триггер | Что грузит | Бюджет |
|------|---------|------------|--------|
| **Tier 1** | Сильное совпадение с burden ИЛИ явный pattern ID | Подобранная цепочка маршрута ИЛИ одна секция | 800–1500 токенов |
| **Tier 2** | Сигнал FPF есть, совпадения burden нет (или слабое) | Динамическая цепочка через keyword + FAISS | ~2000 токенов (вкл. ревьюер) |
| **Tier 3** | Совпадают несколько burden (пересекающая задача) | Core первичного маршрута + семантическое дополнение | ~2500 токенов (вкл. ревьюер) |

Выбор яруса также определяет глубину конвейера — см. [pipeline-depth](pipeline-depth.md). Простые поиски Tier 1 пропускают ревьюера; Tier 2 и Tier 3 всегда его подключают.

В формате вывода классификатора поле `TIER` содержит выбранное значение (1, 2, 3), а поле `PIPELINE` говорит, запускается ли ревьюер.

## Заметка о терминологии

"Три яруса" в этом проекте — о **стратегии загрузки**, а не о шкале "лучше/хуже". Запрос Tier 2 не "лучше" Tier 1 — он просто обрабатывается иначе, потому что подобранного маршрута не нашлось. Если семантический поиск со временем покажет, что какая-то задача достойна собственного маршрута, она переходит из Tier 2 в Tier 1 через создание маршрута, а не через улучшение метрики.
