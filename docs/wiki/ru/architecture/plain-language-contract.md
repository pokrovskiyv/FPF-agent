# Контракт обычного языка

## Компоненты

Контракт обычного языка обеспечивается на трёх слоях: Принцип #0 резонера, Check 1 ревьюера (страж жаргона) и лексические правила, порождаемые `build_lexical`.

| Слой | Что делает |
|------|-----------|
| Принцип #0 резонера | Каждый ответ на языке пользователя; нулевая терминология FPF по дизайну |
| Check 1 ревьюера | Сканирует вывод на запрещённые термины и переписывает нарушения |
| `sections/lexical-rules.md` | Обязательные замены (например, "axis" → Characteristic, "metric" как существительное → Measure/Score) применяются резонером внутренне |

## Поток данных

```
сообщение пользователя (обычные слова)
      │
      ▼
Классификатор (внутренние метки FPF, пользователю не показываются)
      │
      ▼
Ретривер грузит паттерны FPF (агентам видна полная терминология)
      │
      ▼
Резонер:
  - внутренне читает lexical-rules.md
  - применяет структуру паттерна к задаче пользователя
  - пишет на языке пользователя
  - ноль FPF-терминов на выходе
      │
      ▼  (для Tier 2/3)
Ревьюер:
  - сканирует вывод на запрещённые токены
  - переписывает нарушения на обычный язык
  - возвращает STATUS: PASS | CORRECTED
      │
      ▼
пользователь (обычные слова)
```

## Решения

- **Невидимая инфраструктура.** Аналогия из `agents/fpf-reasoner.md`: "Вы — GPS. Внутри используете алгоритм Дейкстры. Пользователю говорите 'поверните направо через 200 метров.' Вы не говорите 'применяется алгоритм кратчайшего пути на взвешенном графе.'"
- **Список запрещённых терминов не исчерпывающий.** Ревьюер инструктирован помечать всё, что *звучит* как жаргон фреймворка — не только явно перечисленное. Это позволяет расширять фреймворк без правки списка.
- **Примеры переписываний в промпте ревьюера.** "Using U.Commitment deontic objects..." → "Here are the obligations this creates...". Это калибровка того, что означает "обычный язык" в данном проекте.
- **Лексические правила — это Part K спеки.** Словарь замен встроен в `FPF-Spec.md` и извлекается `build_lexical.py` — отдельно список запрещённых терминов не поддерживается.
- **Не подлежит обсуждению, по CLAUDE.md.** Каждый коммит и изменение кода в проекте обязан сохранять контракт. Нарушения сразу видны в выводе ревьюера или в smoke-тестах.

## Пример (из CLAUDE.md)

Термины, которые НИКОГДА не должны появляться в пользовательском выводе: `holon`, `bounded context`, `episteme`, `transformer quartet`, `CharacteristicSpace`, pattern ID (`A.6`, `E.17`), `U.*`, аббревиатуры фреймворка (`F-G-R`, `NQD`, `DRR`, `UTS`, ...), мета-ссылки ("according to FPF", "the framework"), термины лексического долга `axis`/`dimension`/`metric` (как существительное).
