# build_routes

> Источник: `scripts/build_routes.py`

## Назначение

Генерирует 10 файлов маршрутов в `sections/routes/` из жёстко заданной константы `ROUTES`. Каждый маршрут кодирует подобранную вручную цепочку секций под одну пользовательскую задачу — например, "команды не понимают друг друга" → маршрут 1 → `[A.1.1, A.15, A.15.2, ...]`. Маршруты работают как кэш поверх семантического поиска: если задача пользователя совпадает с известной, классификатор выбирает маршрут напрямую, минуя полный поиск.

## Интерфейс

| Функция | Сигнатура | Что делает |
|---------|-----------|-----------|
| `build_route_file` | `(route: dict, metadata: dict) -> str` | Формирует markdown одного маршрута: заголовок с "When user says" / "What user gets", таблица цепочки (pattern ID, название, путь к файлу, флаг core) и подвал со стратегией загрузки |
| `main` | `() -> None` | Точка входа argparse (`--metadata`, `--output`); пишет по файлу на маршрут |

На уровне модуля константа `ROUTES` — ручной список из 10 словарей. Каждый содержит `id`, `slug`, `user_says`, `user_gets`, `chain` (упорядоченные pattern ID) и `core` (подмножество для минимальной загрузки).

## Алгоритм

1. Загружаем `sections/metadata.json`, чтобы сопоставить pattern ID с названиями и путями к файлам.
2. Для каждого из 10 маршрутов строим markdown-таблицу: колонка Core помечена `YES` только для паттернов из `route['core']`.
3. Пишем файлы с именами `route-{id}-{slug}.md` в `sections/routes/`.
4. Печатаем по одной строке подтверждения на файл.

Файлы маршрутов — основной актив Tier 1. Они позволяют ретриверу полностью пропускать семантический поиск для распространённых задач.

## Зависимости

**Импорты:** `argparse`, `json`, `pathlib.Path` — только стандартная библиотека.

**Где используется:** вызывается из `scripts/rebuild_all.sh` (шаг 6). Читает вывод [build_metadata](build_metadata.md); созданные файлы читает ретривер во время работы.
